# 医疗实施助理常见现场故障处理手册

这份手册面向医院现场实施工作，目标不是替代开发或运维，而是帮助你快速判断问题在哪一层、先做什么、怎么和相关角色协同。

## 使用原则

- 先确认影响范围，再动手处理。
- 先保留现场，再做低风险操作。
- 先排查链路，再判断责任归属。
- 能截图、记日志时间、记报错内容的，尽量第一时间保留。

## 场景 1：查询病人信息很慢

### 常见表现

- 医生站或护士站打开病人信息转圈很久
- 门诊挂号查询明显变慢
- 某个页面偶发超时，刷新后又恢复

### 排查思路

1. 先确认范围：是全院都慢，还是某个科室、某个功能慢。
2. 记录时间点：问题发生时间、用户账号、病人编号、操作路径。
3. 查应用日志：是否有超时、线程阻塞、接口调用失败。
4. 查数据库：是否有慢 SQL、锁等待、全表扫描。
5. 查服务器：CPU、内存、磁盘 I/O 是否高。
6. 查网络：是否只有某个院区、某个终端、某条专线异常。

### 你可以怎么说

“我会先判断问题是全局性能问题还是单功能问题，再按应用、数据库、服务器、网络四层排查，避免一上来就重启服务导致现场信息丢失。”

## 场景 2：HIS 显示发送成功，但 EMR 说没收到

### 常见表现

- HIS 页面提示接口成功
- EMR 侧没有患者数据或诊断数据
- 对接方说他们日志没收到请求

### 排查思路

1. 先拿原始报文核对字段：住院号、患者姓名、诊断是否为空。
2. 查 HIS 发送日志：目标地址、发送时间、返回码、返回报文。
3. 查接口网关或中间件日志：报文是否真正发出。
4. 查网络链路：IP、端口、路由、反向代理是否正常。
5. 查 EMR 接收日志：是否被鉴权失败、字段校验失败、业务规则拒收。

### 关键结论

- “发送成功”有时只是 HIS 本地发送动作成功，不等于对方业务入库成功。
- 排查接口问题一定要看“报文 + 日志 + 链路”三个对象，不能只听口头反馈。

## 场景 3：Tomcat 进程不在了，系统打不开

### 常见表现

- 浏览器打不开系统页面
- 8080/8090 端口不通
- 页面显示 502、503 或连接失败

### 排查思路

1. `ps -ef | grep tomcat` 看进程是否存在。
2. `ss -lntp | grep 8080` 看端口是否监听。
3. 看 Tomcat 日志：`catalina.out`、应用日志是否有启动报错。
4. 看 Java 进程是否因内存不足退出。
5. 看磁盘是否满、日志是否打爆、JVM 参数是否异常。

### 处理原则

- 不要一开始只会重启。
- 先确认是“进程没了”“端口没起来”还是“进程在但应用初始化失败”。
- 重启前尽量先保留日志和报错时间。

## 场景 4：磁盘 100% 打满，系统崩溃

### 常见表现

- 系统无法登录
- 数据库或 Tomcat 自动退出
- 日志写不进去
- Linux 执行命令卡顿甚至报错

### 排查思路

1. `df -h` 先看哪个分区满了。
2. `du -sh /var/log/*` 或 `du -sh /*` 找大目录。
3. 优先检查日志目录、备份目录、临时文件目录。
4. 定位是否为单个异常日志文件暴涨。

### 快速清理原则

- 优先删确认无用的旧日志、旧压缩包、过期临时文件。
- 不直接删数据库文件、业务上传文件、配置文件。
- 删除前尽量先备份关键日志，保留排障证据。

### 常用命令

```bash
df -h
du -sh /var/log/*
find /var/log -type f -name "*.log" -mtime +30
find /var/log -type f -name "*.log" -mtime +30 -delete
```

### 你可以怎么说

“我会先定位满的是哪个分区，再定位是哪个目录、哪类文件导致占满，优先清理可确认无用的旧日志和临时文件，避免误删核心业务数据。”

## 场景 5：数据库更新或删除前怕误操作

### 风险点

- `UPDATE`、`DELETE` 条件写错
- 忘写 `WHERE`
- 影响行数超预期

### 安全做法

1. 先用同样条件写一遍 `SELECT`，确认会影响哪些数据。
2. 先备份相关记录。
3. 尽量在事务里执行。
4. 执行后立即核对影响行数。
5. 重要变更让同事复核。

### 你可以怎么说

“生产环境做删改前，我会先 `SELECT` 验证范围，再备份，再放进事务执行，避免误伤真实患者数据。”

## 场景 6：用户说“昨天还能用，今天突然不行了”

### 排查切入点

- 今天是否刚发版
- 是否改过配置
- 是否换过证书、IP、账号权限
- 是否有定时任务或清理任务在凌晨运行
- 是否是数据库空间、磁盘空间、授权到期问题

### 经验判断

这种问题常见于：

- 凌晨作业失败
- 服务自动重启后配置未加载
- 证书过期
- 账号锁定
- 接口对方地址变更

## 场景 7：用户反馈“只有我这台电脑不行”

### 排查思路

1. 先判断是不是单终端问题。
2. 换浏览器、换电脑、换账号做对比。
3. 查本机网络、浏览器缓存、证书、hosts、权限控制。
4. 如果同账号换电脑正常，问题大概率在终端环境。
5. 如果同电脑换账号异常消失，问题可能在账号权限。

## 场景 8：实施现场如何和开发、网工、DBA 协作

### 实施该做什么

- 把问题现象描述清楚
- 收集日志时间点、报错内容、影响范围
- 提供复现路径
- 说明已经排除过哪些因素

### 不要怎么做

- 只说“系统坏了”
- 不给时间点、不留截图
- 不区分网络问题、数据库问题、程序问题

### 高质量提单示例

“2026-03-11 10:28，门诊收费处 3 台机器无法查询患者挂号信息，影响收费功能。HIS 页面提示超时，应用日志同时间段有数据库连接超时信息，数据库主机能 ping 通，其他模块暂时正常，怀疑为查询接口或数据库会话异常。”

## 面试高频总结

如果面试官问现场故障排查，建议你统一用这个表达框架：

1. 先确认影响范围。
2. 再保留现场信息。
3. 然后分层排查：应用、数据库、服务器、网络、接口链路。
4. 优先做低风险操作。
5. 最后推动相关角色一起闭环。

## 一句话记忆法

医疗实施不是“哪里坏了就重启哪里”，而是“先定位、再协同、后处理、能复盘”。
